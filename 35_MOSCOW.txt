# MoSCoW Method — Приоритизация требований

**Автор:** Дай Клегг (Oracle), 1994. Популяризован Agile/DSDM
**Тип:** Приоритизация фич, задач, требований
**Сложность:** ★★☆☆☆ (базовая)


## 📖 ИСТОРИЯ СОЗДАНИЯ

1994 год. Дай Клегг, консультант Oracle UK, разрабатывает метод приоритизации требований для RAD (Rapid Application Development). Название MoSCoW — игра слов: Moscow и аббревиатура из первых букв категорий.

Must have (обязательно), Should have (должно быть), Could have (может быть), Won't have (не будет в этой версии). Метод становится стандартом Agile и DSDM.

## 👤 ОБ АВТОРЕ

**Дай Клегг** — британский консультант, соавтор методологии DSDM (Dynamic Systems Development Method).

## КОГДА ПРИМЕНЯТЬ
- MVP: определение минимального объёма для запуска
- Ограниченный бюджет/сроки — что делать в первую очередь
- Спор stakeholder'ов о приоритетах
- Планирование спринта или релиза

## СУТЬ МЕТОДА
MoSCoW = 4 категории приоритетов:

- **M — Must Have (Обязательно):** критично для запуска. Без этого продукт не работает. Обычно ≤60% объёма.
- **S — Should Have (Нужно):** важно, но запуск возможен без этого. Будет в следующей итерации.
- **C — Could Have (Можно):** желательно, но не обязательно. При наличии ресурсов.
- **W — Won't Have (Не сейчас):** не в этом релизе. Возможно, никогда.

## АЛГОРИТМ АНАЛИЗА

### Шаг 1. Составь список требований
Все фичи, задачи, user stories. Без ограничений, полный список.

### Шаг 2. Разнеси по категориям
Для каждого требования задай вопрос: «Что будет, если мы ЭТО НЕ сделаем?»
- Продукт не работает → Must
- Серьёзное неудобство, но работает → Should
- Приятно иметь → Could
- Без этого можно → Won't

### Шаг 3. Проверь баланс
Must должно быть ≤60% трудозатрат. Если Must >60% — нужно пересмотреть.

### Шаг 4. Согласуй со стейкхолдерами
Обсуди с заказчиком: согласен ли он с категориями?

## ФОРМАТ ВЫВОДА

### Приоритизация требований
| # | Требование | MoSCoW | Обоснование |
|---|-----------|--------|-------------|
| 1 | Функция A | Must | Без неё нет ценности |
| 2 | Функция B | Should | Важно, но можно v2 |

### Баланс
- Must: N% от объёма — норма/перебор
- Should: N%
- Could: N%
- Won't: N%

### План MVP
Must-требования: [список из N пунктов]
Оценка сроков: [время]
